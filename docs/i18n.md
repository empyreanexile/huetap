# Translations

HueTap ships with two starter locales — **`en_US`** (American English) and **`en_GB`** (British English) — that sit alongside a canonical English template. Every locale is a small ARB file in [`lib/l10n/`](../lib/l10n). Contributing a new language is a copy-edit plus a PR.

## How the files fit together

| File | Role |
|---|---|
| [`lib/l10n/app_en.arb`](../lib/l10n/app_en.arb) | **Template.** Canonical English source of truth. Every string lives here with its `@key` metadata (description, ICU placeholders). Used as the fallback when a locale is missing a key. |
| [`lib/l10n/app_en_US.arb`](../lib/l10n/app_en_US.arb) | US English variant. Starter example — mirrors the template. |
| [`lib/l10n/app_en_GB.arb`](../lib/l10n/app_en_GB.arb) | UK English variant. Demonstrates a regional override (`"e.g. Nightstand"` → `"e.g. Bedside table"`). |

At build time, `flutter gen-l10n` reads every `app_*.arb` file and emits typed delegates into [`lib/l10n/gen/`](../lib/l10n/gen). If a locale omits a key, the generator falls back through the locale chain (`en_GB` → `en` template), so regional variants only need to spell out the strings that differ.

## Contributing a translation

1. Find your locale code. BCP 47 language tags: `es` for Spanish, `fr` for French, `de_CH` for Swiss German, `pt_BR` for Brazilian Portuguese.
2. Copy [`lib/l10n/app_en.arb`](../lib/l10n/app_en.arb) to `lib/l10n/app_<locale>.arb` (e.g. `app_fr.arb`, `app_pt_BR.arb`).
3. Change the top-level `"@@locale"` value to match your filename suffix.
4. Translate every `value`. **Leave the keys and ICU placeholders alone.** The `@key` metadata blocks in the template are optional in translation files — they're there for translators to read, not to ship.
5. Run `flutter gen-l10n` locally to regenerate the delegates. CI will catch it if you forget.
6. Run `flutter analyze` and `flutter test`.
7. Submit a PR. Review focuses on completeness and technical correctness — linguistic quality is the translator's call.

### Regional variants

If your locale has a well-established regional split (e.g. `es_MX` vs `es_ES`, `pt_BR` vs `pt_PT`), follow the `en_US` / `en_GB` pattern:

- `app_es.arb` — the broad Spanish template (whichever dialect you consider neutral).
- `app_es_MX.arb` — Mexico-specific overrides, only the strings that differ.
- `app_es_ES.arb` — Spain-specific overrides, only the strings that differ.

You don't need all three to land in one PR. Start with the one you know best; others can follow.

## ICU placeholders

Strings that interpolate runtime values use ICU syntax. Example from the template:

```arb
"pairPollLinkButtonDescription": "Push the large round button on top of your Hue bridge at {ip}.",
"@pairPollLinkButtonDescription": {
  "description": "Detailed instruction for the link-button step, including the bridge's IP.",
  "placeholders": {
    "ip": { "type": "String", "example": "192.168.1.10" }
  }
}
```

Translations must preserve `{ip}` (or whatever the placeholder is called) verbatim.

## Plural forms

If a string pluralises, the English template uses ICU plural syntax. For example:

```arb
"discoveryStatusFoundMany": "Found {count} bridges"
```

Today the app handles the `=1` case with a separate key (`discoveryStatusFoundOne`) rather than inline ICU plurals, because Flutter's generator produces cleaner APIs that way. If your language has more plural categories (Polish, Russian, Arabic, etc.), either duplicate the split-key pattern or open an issue — we're happy to migrate specific strings to inline ICU plurals where it matters.

## What not to translate

- Placeholder values like `{ip}`, `{sceneName}`, `{count}`.
- Proper nouns: *HueTap*, *Philips Hue*, *NTAG213/215/216*, *CLIP v2*, *Sentry*.
- The `"@@locale"` metadata at the top of the ARB file (set it to match your filename).

## Registering the locale

New locales are picked up automatically by `flutter gen-l10n` — no manual `supportedLocales` editing needed. The MaterialApp in [`lib/main.dart`](../lib/main.dart) consumes `AppLocalizations.supportedLocales`, which the generator keeps in sync with whichever ARB files are present.

## Keeping translations in sync

When English strings change, existing translations go stale. We don't currently have tooling that flags stale translations; contributors are welcome to submit updates when they notice drift. Missing keys fall back to the English template, so a partially stale translation still renders a usable UI.
