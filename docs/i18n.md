# Translations

> **Stub.** English is the only language at launch. ARB infrastructure is in place for contributions.

HueTap uses Flutter's standard localization stack: `flutter_localizations` + `intl` + ARB files in [`lib/l10n/`](../lib/l10n).

## Contributing a translation

1. Find your locale code on [this list](https://developers.google.com/admin-sdk/directory/v1/languages) (e.g. `es` for Spanish, `fr` for French, `de_CH` for Swiss German).
2. Copy [`lib/l10n/app_en.arb`](../lib/l10n/app_en.arb) to `lib/l10n/app_<locale>.arb`.
3. Translate every `value`. **Leave the keys and `@@locale` metadata alone.** Preserve ICU placeholders like `{bridgeName}` exactly.
4. Run `flutter gen-l10n` locally to regenerate the delegates (optional; CI will catch it).
5. Run `flutter analyze` and `flutter test`.
6. Submit a PR. Review focuses on completeness and technical correctness — not linguistic quality.

## ICU placeholders

Some strings interpolate runtime values like bridge names or scene names. These use ICU:

```arb
"bridgeUnreachable": "{bridgeName} unreachable. Check your wifi.",
"@bridgeUnreachable": {
  "placeholders": {
    "bridgeName": { "type": "String" }
  }
}
```

Translations must keep `{bridgeName}` (or whatever the placeholder is called) unchanged.

## Plural forms

If a string pluralizes (e.g. "N bridges need re-pairing"), the English ARB uses ICU plural syntax:

```arb
"bridgesNeedRepairing": "{count, plural, =1{1 bridge needs re-pairing} other{{count} bridges need re-pairing}}"
```

Some languages have more plural categories (e.g. Polish has three). Add them as needed.

## What not to translate

- Placeholder values like `{bridgeName}`, `{count}`
- Proper nouns: *HueTap*, *Philips Hue*, *NTAG215*, *CLIP v2*, *Sentry*
- The `@@locale` metadata at the top of the ARB

## Keeping translations in sync

When English strings change, translations go stale. We don't currently have tooling that flags stale translations; contributors are welcome to submit updates when they notice drift.
